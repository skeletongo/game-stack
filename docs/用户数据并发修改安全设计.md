# 用户数据并发修改安全设计

## 问题

玩家状态数据的修改有三个来源：

1. **外部主动消息** — 客户端请求、GM 指令等外部触发的操作，通过路由到达
2. **内部消息队列消息** — 跨模块事件通知，如匹配成功、战斗结算、邮件到达等，通过 EventBus 投递
3. **跨节点 gRPC 调用** — 其他节点通过 gRPC 直连调用本节点服务，运行在 gRPC goroutine 上

`StatefulRoute` 只保证请求路由到同一节点，**不保证串行执行**。同一节点的多个 goroutine 可以同时处理同一玩家的不同请求。

假设玩家当前金币 100，同时发起"升级"和"买装备"两个操作：

```
时间轴 →

goroutine A（升级）：
  读玩家数据 → Gold=100
    计算新级 → 经验够了，Gold+=50
      写回 → UpdatePlayer({Gold=150, ...})

goroutine B（买装备）：
    读玩家数据 → Gold=100
      扣金币 → Gold-=60
        写回 → UpdatePlayer({Gold=40, ...})
        
结果：Gold=40
期望：Gold=90（100+50-60）  ← A 的 50 增量被 B 覆盖丢了
```

**原因**：两个 goroutine 都读到了旧值（100），各自基于旧值计算，后写入的覆盖先写入的。

跨模块场景更严重——shop 扣金币 + inventory 加道具跨两个 Store，读-改-写不在同一个锁内，组合操作完全不原子。

## 解决方案：Actor 串行化

每个在线玩家拥有一个 `PlayerActor`。**所有玩家状态修改操作，无论触发来源，统一通过 Actor 串行执行。**

```
客户端请求 → Node(StatefulRoute) → RouteToActor → Actor.mailbox → handler → ctx.Response()
                                                          ↑
EventBus 事件 → EventHandler → actor.Invoke(uid, fn) ────┤
                                                          │
gRPC 调用 → server handler → actor.Invoke(uid, fn) ──────┘
```

### 为什么用 Actor 而非 Store 锁

| 方案 | 保护范围 | 问题 |
|------|----------|------|
| Store 层 per-player 锁 | 单个 Store 操作的 RMW | 跨 Store 操作不原子（扣金币 + 加道具跨两个 Store） |
| 全局锁 | 所有玩家 | A 的操作阻塞 B，完全违背多核并行 |
| **Actor 串行化** | 同一玩家的所有操作 | 不同玩家完全并行，同一玩家完全串行 |

### 何时可以不走 Actor

| 场景 | 原因 |
|------|------|
| 纯查询操作（无状态修改） | 无并发风险 |
| 全局共享数据（排行榜、商店列表等不归属特定玩家的数据） | 不存在"同一玩家"的竞争 |
| 无状态操作（登录、注册） | 此时 Actor 尚未创建 |

## 两种使用模式

### 模式 1：RouteToActor（客户端请求 → 同步响应）

整个 handler 在 Actor goroutine 中执行，跨 Store 操作天然串行安全。

**模块 Init 时注册两步：**

```go
func (m *exampleModule) Init(proxy *node.Proxy) error {
    impl := newImpl(o.store)

    // ① Node 路由用 RouteToActor 转发
    proxy.AddRouteHandler(stack.RouteExampleXxx,
        actor.RouteToActor(actor.KindPlayer),
        stack.StatefulAuthorizedRoute,
    )

    // ② 注册 Actor 路由初始化器（每个 Actor Spawn 时自动应用）
    actor.RegisterRouteInitializer(func(act *node.Actor) {
        act.AddRouteHandler(stack.RouteExampleXxx, impl.handleXxxActor)
    })
}
```

**Actor handler：**

```go
func (i *impl) handleXxxActor(ctx node.Context) {
    req := &protocol.XxxRequest{}
    ctx.Parse(req)

    playerSvc.DeductGold(ctx.UID(), price)
    inventorySvc.AddItem(ctx.UID(), itemID, count)

    stack.RespondOK(ctx)
}
```

**消息流：**

```
客户端 → Gate → Node(StatefulRoute) → RouteToActor → Actor.mailbox → handler → ctx.Response()
```

### 模式 2：Invoke（消息队列触发 → fire-and-forget）

跨模块事件触发，不需要同步响应：

```go
func (i *impl) handleMatchFound(ctx node.Context) {
    uid := ctx.UID()
    actor.Invoke(i.proxy, uid, func() {
        playerSvc.AddExp(uid, 100)
        questSvc.UpdateProgress(uid, questMatchWin, 1)
    })
}
```

```
EventBus 事件 → EventHandler → actor.Invoke(uid, fn) → Actor.fnChan → dispatch goroutine 串行执行
```

## 安全机制

### 断线截断

`KillPlayer` 后 mailbox 和 fnChan 关闭，所有排队消息/Invoke 被丢弃。杜绝"断线幽灵消息"：旧节点上的待处理操作不会在玩家已迁移后继续执行。

### 归属权校验

`RouteToActor` 和 `Invoke` 在投递前通过 `LocateNode` 校验玩家是否仍绑定在本节点。如果 Disconnect 事件丢失导致旧节点 Actor 残留，第一条到达的消息就能触发自毁：

```
玩家在 Node A → 断线（Disconnect 丢失）
    → 重连到 Node B → BindNode → Locator 更新
    ↓
Node A 的 Actor 残留，下一条消息到达：
  RouteToActor → LocateNode → nid != myID → Kill + 拒绝请求
```

- `LocateNode` 优先读本地缓存（Redis Pub/Sub 实时同步），零额外延迟
- Disconnect 丢失但 Pub/Sub 正常 → 第一条消息即自毁
- 两者都丢失属于集群级灾难，玩家的消息路由本身也已不可用，无需额外防御

### Grace Period 并发安全

断线瞬间 `KillPlayer` 关闭 mailbox + fnChan。Grace Period 期间 Actor 不存在，不会有新消息进入。`doCleanup` 清理前通过 `LocateGate` 确认无活跃连接。重连时先取消 cleaner 定时器，再 `SpawnPlayer`。

详见 `docs/用户延迟登出设计.md` 和 `docs/用户绑定节点设计.md`。

### gRPC 写操作

gRPC handler 运行在 gRPC goroutine 上，不在 Actor 内。如果 gRPC 方法需要修改玩家状态，必须通过 `actor.Invoke` 投递到 Actor：

```go
func (s *server) UpdatePlayer(ctx context.Context, req *UpdatePlayerReq) (*UpdatePlayerResp, error) {
    // gRPC goroutine — 不能直接修改玩家数据
    // 通过 Actor 串行化
    actor.Invoke(s.proxy, req.Uid, func() {
        s.svc.AddExp(req.Uid, req.Exp)
    })
    return &UpdatePlayerResp{}, nil
}
```

纯查询（如 `GetPlayer`）不受此限制，可直接在 gRPC handler 中执行。

# 用户绑定节点设计

## 概述

玩家在集群中需要"黏着"到某个特定节点，因为玩家内存数据（状态、缓存）只存在于该节点。绑定机制通过 Redis Locator + 有状态路由实现。

## 生命周期

```
首次登录 (无状态路由)
    │
    ├── Gate 负载均衡器随机选取一个可用 Node
    │
    └── Node：BindGate(uid) + BindNode(uid) + SpawnPlayer
            │
            │  BindGate: 将 WebSocket 连接与 UID 关联
            │  BindNode: HSET due:locate:user:{uid}:node game-node {node-id}
            │
            ▼
        已绑定状态
            │
            │  所有 StatefulRoute 的消息：
            │  Gate 先查 Locator 找玩家绑定的节点，再定向投递
            │
            ▼
        断线 → KillPlayer + playerlife.OnDisconnect（30s Grace Period）
            │
            │  token 和节点绑定保留（等待重连）
            │
            ├── 30s 内重连 → 见下方「重连流程」
            │
            └── 30s 超时 → doCleanup
                    │
                    ├── LocateGate 确认无活跃连接
                    └── UnbindNode（最后解绑）
```

## 重连流程

断线后 Grace Period 期间，token 和节点绑定保留。重连处理分三种情况：

```
客户端重连 → 发送 Login 请求 → 无状态路由 → 随机落到 Node X
    │
    └── Node X：验证密码 → LocateNode(uid)
            │
            ├── 无绑定（Grace Period 已过期 or 首次登录）
            │     → BindNode(uid)（绑定到当前节点）
            │     → SpawnPlayer
            │
            ├── 绑定在本节点（恰好落回原节点）
            │     → playerlife.OnLogin（取消定时器）
            │     → Actor 不存在则 SpawnPlayer（恢复数据）
            │
            └── 绑定在其他节点（请求落到了新节点）
                  → 不 BindNode（保留旧节点绑定）
                  → 不 SpawnPlayer
                  → 只 BindGate + 签发 token + 返回响应
                        │
                        │  BindGate 触发 cluster.Connect 全集群广播
                        ▼
                  旧节点收到 Connect 事件：
                    → playerlife.OnLogin（取消定时器）
                    → Actor 不存在则 SpawnPlayer（恢复数据）
                        │
                        ▼
                  后续 StatefulRoute 消息 → Gate 查 Locator
                    → 仍投递到旧节点 → 缓存命中
```

**关键设计：**
- 重连落到新节点时不覆盖节点绑定，旧节点通过广播的 `cluster.Connect` 事件感知并恢复 Actor
- 后续所有有状态路由仍走旧节点，数据零丢失
- 30s 内未重连 → `doCleanup` 通过 `LocateGate` 确认无活跃 Gate 连接后解绑节点

## 路由类型

| 路由选项 | 说明 | 登录后消息流向 |
|----------|------|---------------|
| 无选项 | 无状态路由 | Gate → 负载均衡 → 任意 Node |
| `StatefulRoute` | 有状态路由 | Gate → 查 Locator → 玩家绑定 Node |
| `AuthorizedRoute` | 需认证 | 必须携带合法 UID |
| `StatefulAuthorizedRoute` | 有状态 + 需认证 | 两者结合，绝大多数游戏操作用此 |

game-stack 中只有登录和注册是无状态的，其他所有操作都标记了 `StatefulAuthorizedRoute`。

## Locator 机制

Redis Locator 使用三种 key 结构：

| Key | 类型 | 内容 |
|-----|------|------|
| `due:locate:user:{uid}:gate` | String | 网关 ID |
| `due:locate:user:{uid}:node` | Hash | field=节点名称, value=节点实例 ID |
| `due:locate:cluster:{kind}:event` | Pub/Sub | 绑定/解绑事件广播 |

所有 Node 和 Gate 实例通过订阅 Pub/Sub channel 实时感知定位变更，更新本地缓存。没有 Locator，不同进程无法知晓玩家的节点归属。

`doCleanup` 中使用 `LocateGate`（而非 `AskNode`）检查玩家是否重连，因为 Grace Period 期间节点绑定保留，`AskNode` 始终返回 true，无法区分"未曾重连"和"已重连"。

## 解绑安全：Lua 脚本

`UnbindNode` 不是简单的 `HDEL`，而是通过 Lua 脚本原子执行：

1. `HGET key field` 获取当前绑定的 nid
2. 与传入的 `ARGV[2]`（本节点 ID）比较
3. 匹配 → `HDEL` 并返回 OK
4. 不匹配 → 拒绝操作

这防止了以下竞态：玩家在旧节点还未执行 UnbindNode 的瞬间，已在新节点完成了 BindNode。旧节点的 Lua 脚本校验发现绑定已被新节点覆盖，拒绝解绑。

## 两阶段生命周期

断线后不立即解绑，而是通过 `module/playerlife.Manager` 管理 Grace Period：

| 阶段 | 时机 | 执行操作 |
|------|------|----------|
| 立即 | 断线瞬间 | `KillPlayer`（杀 Actor，关闭 mailbox） |
| 延迟 | 30s 后未重连 | 确认无活跃 Gate 连接 → `UnbindNode`（最后解绑） |

Grace Period 期间 token 和 node 绑定保留，玩家重连可无缝恢复缓存数据。详见 `docs/用户延迟登出设计.md`。

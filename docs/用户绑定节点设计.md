# 用户绑定节点设计

## 概述

玩家在集群中需要"黏着"到某个特定节点，因为玩家内存数据（状态、缓存）只存在于该节点。绑定机制通过 Redis Locator + 有状态路由实现。

## 生命周期

```
首次登录 (无状态路由)
    │
    ├── Gate 负载均衡器随机选取一个可用 Node
    │
    └── Node：BindGate(uid) + BindNode(uid) + SpawnPlayer
            │
            │  BindGate: 将 WebSocket 连接与 UID 关联
            │  BindNode: HSET due:locate:user:{uid}:node game-node {node-id}
            │
            ▼
        已绑定状态
            │
            │  所有 StatefulRoute 的消息：
            │  Gate 先查 Locator 找玩家绑定的节点，再定向投递
            │
            ▼
        普通断线
            │
            │  token 由 auth 决定是否保留；节点绑定不变
            │
            └── 后续重连或有状态消息仍路由到已绑定节点
```

## 重连流程

普通断线不会解绑节点。重连处理分三种情况：

```
客户端重连 → 发送 Login 请求 → 无状态路由 → 随机落到 Node X
    │
    └── Node X：验证密码 → LocateNode(uid)
            │
            ├── 无绑定（首次登录或绑定节点已失效）
            │     → BindNode(uid)（绑定到当前节点）
            │     → SpawnPlayer
            │
            ├── 绑定在本节点（恰好落回原节点）
            │     → Actor 不存在则 SpawnPlayer
            │
            └── 绑定在其他节点（请求落到了新节点）
                  → 不 BindNode（保留旧节点绑定）
                  → 不 SpawnPlayer
                  → 只 BindGate + 签发 token + 返回响应
                        │
                        ▼
                  后续 StatefulRoute 消息 → Gate 查 Locator
                    → 仍投递到旧节点 → RouteToActor 按需 SpawnPlayer
```

**关键设计：**
- 重连落到新节点时不覆盖节点绑定，后续有状态路由仍走旧节点。
- Actor 不再跟随连接生命周期销毁，缺失时由登录或 `RouteToActor` 按需创建。
- 普通断线不解绑节点；后续唯一修改已有节点归属的能力是显式玩家迁移流程。

## 路由类型

| 路由选项 | 说明 | 登录后消息流向 |
|----------|------|---------------|
| 无选项 | 无状态路由 | Gate → 负载均衡 → 任意 Node |
| `StatefulRoute` | 有状态路由 | Gate → 查 Locator → 玩家绑定 Node |
| `AuthorizedRoute` | 需认证 | 必须携带合法 UID |
| `StatefulAuthorizedRoute` | 有状态 + 需认证 | 两者结合，绝大多数游戏操作用此 |

game-stack 中认证管理类路由（注册、登录、登出、刷新令牌）是无状态路由；修改玩家状态的游戏操作使用 `StatefulAuthorizedRoute`。

## Locator 机制

Redis Locator 使用三种 key 结构：

| Key | 类型 | 内容 |
|-----|------|------|
| `due:locate:user:{uid}:gate` | String | 网关 ID |
| `due:locate:user:{uid}:node` | Hash | field=节点名称, value=节点实例 ID |
| `due:locate:cluster:{kind}:event` | Pub/Sub | 绑定/解绑事件广播 |

所有 Node 和 Gate 实例通过订阅 Pub/Sub channel 实时感知定位变更，更新本地缓存。没有 Locator，不同进程无法知晓玩家的节点归属。

## 迁移安全

普通断线、业务登出和 token 失效都不修改节点绑定。这样可以避免连接生命周期和数据归属互相影响。

已有节点归属需要改变时，必须走显式玩家迁移流程。迁移流程应负责：

1. 暂停或拒绝迁移中的玩家写操作。
2. 在源节点持久化或转移玩家状态。
3. 原子更新 Locator 中的节点绑定。
4. 在目标节点创建 Actor 并恢复状态。
5. 清理源节点残留 Actor。

迁移流程是独立能力，不由普通 Disconnect 事件触发。

# 用户延迟登出设计

## 问题背景

玩家断线后，有两种处理策略：

1. **立即解绑**：断开瞬间解绑节点，玩家重连时重新选择节点
2. **延迟解绑**：保留一段时间，允许玩家无感重连到原节点

立即解绑的问题是：网络抖动、切换 Wi-Fi、短暂信号丢失等情况非常常见。若每次断线都立即解绑，重连后可能落到新节点，导致 Actor 和仓储缓存频繁重建。

延迟解绑的核心挑战：

- **消息队列竞态**：旧节点上的 Actor mailbox 可能还有排队消息未处理。如果断线后继续执行，会修改玩家状态
- **重连到新节点**：如果在 Grace Period 期间玩家重新登录到另一个节点，旧节点需要感知重连并恢复 Actor

## 解决方案：playerlife 两阶段生命周期

玩家连接生命周期由 `module/playerlife` 管理。`playerlife` 是运行时组件，不负责 Repository 的内存释放策略。

### 阶段 1：立即操作

断线瞬间执行，不可延迟：

| 操作 | 原因 |
|------|------|
| `actor.KillPlayer` | 关闭 mailbox，丢弃所有排队消息，杜绝幽灵消息 |

Actor 在断线瞬间被杀死，确保不会再有排队消息修改玩家数据。这是解决消息竞态的关键。

Grace Period 期间 **token 和节点绑定保留**。断线不是业务登出，不应清 token，也不应删除聚合数据。

### 阶段 2：延迟解绑

通过 `playerlife.Manager` 管理：

```
断线 → OnDisconnect(uid)
         │
         ├── KillPlayer
         ├── 启动 30s 定时器
         ├── 30s 内重连 → OnLogin(uid) 取消定时器，并在归属节点恢复 Actor
         │
         └── 30s 超时 → cleanup(uid)
                          │
                          ├── LocateGate 确认无活跃连接
                          │   ├── 查询失败 → 延迟重试
                          │   ├── 有连接 → 跳过解绑（已重连）
                          │   └── 无连接 → 继续
                          │
                          ├── LocateNode 确认本节点仍拥有玩家归属权
                          │
                          └── UnbindNode（最后解绑）
```

Repository 自己决定是否保留内存、何时释放缓存、如何从持久层回源。`playerlife` 不调用 Repository，也不关心仓储是纯内存、缓存仓储还是持久仓储。

### 安全检查

`cleanup` 通过 `LocateGate` 检查玩家是否有活跃 Gate 连接。因为 Grace Period 期间节点绑定保留，不能用 `AskNode` 判断是否重连。

`LocateGate` 查询失败时视为临时故障，进入重试；不能在定位系统异常时继续解绑节点。

`cleanup` 还会通过 `LocateNode` 确认当前节点仍是玩家归属节点。若归属权已丢失，旧节点跳过解绑，避免影响新节点状态。

### 失败重试

`playerlife.Manager` 只对定位检查失败做重试：

| `maxRetries` | 行为 |
|-------------|------|
| `-1` | 不重试，直接解绑 |
| `0` | 一直重试直到定位成功 |
| `>0` | 最多重试 N 次，之后强制解绑 |

默认配置为 30 秒 Grace Period + 最多 3 次重试（`playerlife.Module()` 默认值，可通过 `playerlife.WithDelay()` / `playerlife.WithMaxRetries()` 调整）。

## 代码结构

### Manager（`module/playerlife/manager.go`）

`playerlife` 在 `cmd/node/main.go` 中作为第一个模块注册：

```go
stack.WithModules(
    playerlife.Module(),
    auth.Module(),
    player.Module(),
)
```

主要方法：

```
playerlife.Module(opts...)       → 创建生命周期组件
playerlife.Get()                 → 获取全局 Manager
  .OnDisconnect(uid)             → 启动延迟解绑定时器
  .OnLogin(uid)                  → 取消定时器（重连时）
  .HandleConnect(ctx)            → 集群 Connect 事件处理器
  .HandleDisconnect(ctx)         → 集群 Disconnect 事件处理器
```

Connect/Disconnect 事件由 `playerlife.Module().Init` 自行注册，业务模块不再注册这些事件。

### 业务断线状态

业务模块通常不需要处理 Connect/Disconnect。只有需要断线即时状态更新时，才通过 `stack.AddEventHandler` 注册事件处理器，并在处理器内通过 `LocateNode` 校验当前节点归属权。例如 auth 模块可在断线时 `SetOffline`，但仍保留 token 以支持 Grace Period 内重连。

## Repository 内存管理

Repository 对外不区分内存层或持久层。模块只依赖 `Load` / `Save` / `Delete` 等仓储接口，缓存、TTL、LRU、回源、内存释放都由具体仓储实现自己决定。

约定：

- `Load`：加载聚合，必要时由仓储从持久层回源。
- `Save`：保存聚合，仓储自行决定写缓存、写持久层或 write-through。
- `Delete`：业务删除聚合，不能被断线生命周期调用。

## Grace Period 选择

默认 30 秒，足够覆盖绝大多数网络波动（Wi-Fi 切换、电梯信号丢失等）。可根据业务需求调整，但不应超过 60 秒，太长会让节点绑定保留过久。

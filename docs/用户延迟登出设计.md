# 用户延迟登出设计

## 问题背景

玩家断线后，有两种处理策略：

1. **立即清理**：断开瞬间释放所有内存数据，玩家重连时重新加载
2. **延迟清理**：保留一段时间，允许玩家无感重连

立即清理的问题是：网络抖动、切换 Wi-Fi、短暂信号丢失等情况非常常见，每次断线都触发完整的清理+重建会带来不必要的 CPU 和 IO 开销。

延迟清理的核心挑战：

- **消息队列竞态**：旧节点上的 Actor mailbox 可能还有排队消息未处理。如果延迟期间这些消息继续执行，会修改"即将被清理"的玩家数据，导致新节点上的状态不一致
- **重连到新节点**：如果在 Grace Period 期间玩家重新登录到另一个节点，旧节点需要感知重连并恢复数据

## 解决方案：两阶段清理

### 阶段 1：立即操作

断线瞬间执行，不可延迟：

| 操作 | 原因 |
|------|------|
| `SetOffline` | 标记离线，其他模块可感知 |
| `KillPlayer` (Actor) | 关闭 mailbox，丢弃所有排队消息，杜绝幽灵消息 |

Actor 在断线瞬间被杀死，确保不会再有排队消息修改玩家数据。这是解决"消息竞态"的关键。

**以下操作为延迟执行**（Grace Period 到期时通过 `doCleanup` 执行）：
- 清除 token（auth 模块 `CleanPlayerData`：`account.Logout()` → `repo.Save()`）
- 其他模块的内存数据清理
- `UnbindNode`（所有 `CleanPlayerData` 成功后最后执行）

Grace Period 期间 token 和节点绑定保留，玩家重连可无缝恢复。

### 阶段 2：延迟操作（内存清理）

通过 `PlayerDoneCleaner` 管理：

```
断线 → OnDisconnect(uid)
         │
         ├── 启动 30s 定时器
         │
         ├── 30s 内重连 → OnLogin(uid) 取消定时器，数据保留
         │     （通过 cluster.Connect 广播事件或 Login 落回原节点触发）
         │
         └── 30s 超时 → doCleanup(uid)
                          │
                          ├── 安全检查：LocateGate 确认无活跃连接
                          │   ├── 有连接 → 跳过清理（已重连）
                          │   └── 无连接 → 继续
                          │
                          ├── 遍历所有 CleanablePlayer，调用 CleanPlayerData(uid)
                          │   ├── 全部成功 → UnbindNode
                          │   └── 有失败 → 根据 maxRetries 决定是否延迟重试
                          │         ├── -1 → 不重试，直接 UnbindNode
                          │         ├── 0  → 无限重试直到全部成功
                          │         └── >0 → 最多重试 N 次后强制 UnbindNode
                          │
                          └── UnbindNode（最后解绑，防止清理失败丢数据）
```

### 安全检查

`doCleanup` 通过 `LocateGate` 检查玩家是否有活跃的 Gate 连接。因为 Grace Period 期间节点绑定保留，不能用 `AskNode`（`AskNode` 始终返回 true）。`LocateGate` 直接查询 Gate 层：有 Gate 连接说明玩家已重连，跳过清理。

`time.AfterFunc` 的回调和 `OnLogin` 之间存在天然竞态：

1. `doCleanup` 的 goroutine 已经被调度，但还没拿到锁
2. 同一瞬间玩家重连，`OnLogin` 先拿到锁，删除了定时器
3. `doCleanup` 拿到锁后 `delete(c.timers, uid)` 没事
4. `doCleanup` 释放锁后继续执行 → `LocateGate` 二次确认 → 发现已重连 → 跳过清理

### 失败重试

`CleanPlayerData` 可能因网络抖动、Redis 临时不可用等原因失败。`PlayerDoneCleaner` 支持配置重试策略：

| `maxRetries` | 行为 |
|-------------|------|
| `-1` | 不重试，清理一次后直接解绑 |
| `0` | 一直重试直到全部成功 |
| `>0` | 最多重试 N 次，之后强制解绑 |

默认配置为 30 秒 Grace Period + 最多 3 次重试（`clean.Module()` 默认值，可通过 `clean.WithDelay()` / `clean.WithMaxRetries()` 调整）。

重试期间多个 `CleanablePlayer` 可能部分成功、部分失败，重试时成功的服务会被再次调用。**仓储 Delete 必须幂等**（已删除的聚合再次删除返回 nil），详见 `docs/store设计.md`。

## 代码结构

### PlayerDoneCleaner（`module/clean/player.go`）

清理器已独立为 `module/clean` 公共模块，作为全局单例使用：

```go
// 在需要清理器的模块 Init 中
cleaner := clean.Get()
cleaner.Register(&cleanableAdapter{repo: repo})
```

主要方法：

```
clean.Module(opts...)              → 创建 clean 模块（在 main.go 中注册）
clean.Get()                        → 获取全局清理器单例
  .Register(svc CleanablePlayer)   → 模块注册清理回调
  .OnDisconnect(uid)               → 启动延迟清理定时器
  .OnLogin(uid)                    → 取消定时器（重连时）
  .HandleConnect(ctx)              → 集群 Connect 事件处理器
  .HandleDisconnect(ctx)           → 集群 Disconnect 事件处理器
  .doCleanup(uid)                  → 安全检查 + 重试清理 + 解绑
```

### CleanablePlayer 接口（`module/clean/player.go`）

```go
type CleanablePlayer interface {
    CleanPlayerData(uid int64) error
}
```

需要清理玩家内存数据的模块实现此接口并注册到 `PlayerDoneCleaner`。例如：auth 模块清理 token、player 模块清理内存数据。

### 在模块中的集成

每个模块的 `module.go` 中定义 `cleanableAdapter`，注册到全局清理器：

```go
// module/auth/module.go
func (m *authModule) Init(proxy *node.Proxy) error {
    // ...

    // 连接/断开事件（全集群广播）
    cleaner := clean.Get()
    proxy.AddEventHandler(cluster.Connect, cleaner.HandleConnect)
    proxy.AddEventHandler(cluster.Disconnect, cleaner.HandleDisconnect)

    // 注册清理回调（Grace Period 到期后清除 token）
    cleaner.Register(&cleanableAdapter{repo: repo})
    // ...
}

// cleanableAdapter 适配 AccountRepository → clean.CleanablePlayer。
type cleanableAdapter struct {
    repo domain.AccountRepository
}

func (a *cleanableAdapter) CleanPlayerData(uid int64) error {
    return a.repo.Delete(context.Background(), uid)
}
```

**重要**：`clean.Module()` 必须在 player/auth 等依赖它的模块之前注册到 `stack.WithModules()` 中，确保 `clean.Get()` 调用时已初始化。

## Grace Period 选择

默认 30 秒，足够覆盖绝大多数网络波动（Wi-Fi 切换、电梯信号丢失等）。可根据业务需求调整，但不应超过 60 秒——太长时间会导致内存资源浪费。
